# 3. MVP範囲定義

MVP (Minimum Viable Product) で実装する機能と除外する機能を明確に定義します。

## ✅ MVPに含む機能(優先度順)

### 1. エージェント選択UI (P0 - 最重要)

**実装内容**

- サイドバーにエージェント一覧表示
- 選択中エージェントの名前を左上に表示
- エージェント説明をプラットフォーム上で確認可能

**成功基準**

- エージェントをクリックで選択できる
- 選択中のエージェントがハイライトされる
- デフォルトで汎用チャットが選択されている

**所要時間**: 3時間

### 2. テキストチャット (P0 - 最重要)

**実装内容**

- メッセージ送信
- ストリーミングレスポンス表示
- セッション内会話履歴保持

**成功基準**

- メッセージが送信できる
- AIレスポンスが1文字ずつ表示される
- 過去の会話がセッション内で保持される
- レスポンス速度: 初回トークン < 3秒
- ストリーミング遅延 < 100ms/chunk

**所要時間**: 4時間

### 3. MCPツール統合 (P0 - 最重要)

**実装内容**

- Webスクレイピング機能
- 既存MCP Serverの接続

**成功基準**

- URLを含むメッセージでスクレイピングが実行される
- ツール実行中の状態が表示される
- 取得したWebページ情報が回答に含まれる

**所要時間**: 5時間

### 4. 最低1種のエージェント実装 (P0)

**実装内容**

- デフォルト汎用チャットエージェント
- オプション: スクレイピング特化エージェント

**成功基準**

- 汎用チャットが動作する
- エージェント切り替え < 1秒

**所要時間**: 2時間

## ❌ MVPから除外する機能

### 1. ファイルアップロード機能

**除外理由**

- 実装複雑度が高い
- S3統合がPhase 2で必要
- MVPのコア機能ではない

**Phase 2で実装**

- PDF/TXT/MDアップロード
- S3保存
- コンテキスト統合

### 2. 会話履歴の永続化

**除外理由**

- DBセットアップが必要
- セッションメモリでMVPは十分
- 3日間での実装が困難

**代替手段 (MVP)**

- Python辞書でインメモリ管理
- サーバー再起動で消失(許容)

**Phase 2で実装**

- PostgreSQL RDS
- セッションテーブル
- メッセージテーブル

### 3. エージェント切り替え(会話中)

**除外理由**

- 複雑な状態管理が必要
- MVPではシンプルなフローを優先

**代替手段 (MVP)**

- チャット開始時にエージェント選択
- 選択後は固定

**Phase 3で実装**

- 会話中の切り替えUI
- コンテキスト引き継ぎ

### 4. 複数MCPツール統合

**除外理由**

- 1つのMCPツールで概念検証十分
- 複数ツールの調整に時間がかかる

**代替手段 (MVP)**

- Webスクレイピングのみ

**Phase 2で実装**

- ファイルシステムMCP
- DB接続MCP
- カスタムMCP Server

### 5. ユーザー認証

**除外理由**

- MVPでは単一ユーザー想定
- 認証基盤の構築が大きなタスク

**Phase 3で実装**

- AWS Cognito / Auth0
- ユーザー管理UI
- 権限制御

### 6. クラウドデプロイ

**除外理由**

- MVPはローカル環境で十分
- インフラ構築に時間がかかる

**Phase 2で実装**

- AWS Lambda
- S3 + CloudFront
- RDS

## MVP開発の重点

### 優先順位

1. **動くこと** - 完璧よりも動作を優先
2. **シンプルさ** - 複雑な機能は後回し
3. **検証可能性** - コア機能の検証が目的

### 時間配分

**Day 1 (7時間)**

- プロジェクトセットアップ: 1時間
- バックエンド基盤: 5時間
- 動作確認: 1時間

**Day 2 (7時間)**

- MCP統合: 5時間
- エージェント拡張: 2時間

**Day 3 (7時間)**

- フロントエンド: 5時間
- 統合テスト: 1時間
- ドキュメント: 1時間

### リスク管理

**高リスク項目**

- MCP統合の複雑さ
- ストリーミング実装の難しさ
- LM Studioの安定性

**緩和策**

- 早めに技術検証
- シンプルな実装から開始
- 問題発生時の代替案準備
